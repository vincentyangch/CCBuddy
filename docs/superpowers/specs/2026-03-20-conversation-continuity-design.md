# Conversation Continuity Design

**Date:** 2026-03-20
**Status:** Approved

## Overview

Enable Po to maintain multi-turn conversations by resuming Claude Code SDK sessions. Within a channel, consecutive messages reuse the same SDK session so Po has full conversation history. After a configurable idle timeout (default: 1 hour), a fresh session starts.

Currently, every incoming message spawns a fresh `query()` call with no conversation history — only a prepended `<memory_context>` text block assembled from SQLite. With this change, Po will pass `resume: sdkSessionId` to `query()`, giving Claude the full prior conversation transcript.

## Session Lifecycle

1. **Message arrives** → gateway computes `sessionKey` (see Session Key Strategy)
2. **Lookup** → check `SessionStore` for an active SDK session UUID mapped to this sessionKey
3. **No active session** → call `query({ prompt, options: { sessionId: newUUID } })`. Inject `<memory_context>` (profile + summaries). Store the `sessionKey → sdkSessionId` mapping.
4. **Active session found** → call `query({ prompt, options: { resume: storedUUID } })`. Skip memory context (the SDK session already has conversation history). **Do not set `sessionId`** — `resume` and `sessionId` are mutually exclusive in the SDK.
5. **Idle timeout (default 1 hour)** → `SessionStore.tick()` expires the mapping. Next message creates a fresh session.
6. **Resume failure** → if `resume` throws, catch the error, call `sessionStore.remove()`, and retry as a new session with memory context. Detection: any error thrown when `resumeSessionId` was set on the request is treated as a resume failure (vs. a normal query error which is not retried).

## Session Key Strategy

Session keys determine which messages share a conversation:

- **DMs:** per-user — `userId-platform-dm`. Each user gets their own session.
- **Group channels:** shared — `platform-channelId`. All users in the channel share one session, so Po can reference cross-talk.

**Note:** The existing `buildSessionId()` produces per-user keys (`userName-platform-channelId`). For group channels, the session key must strip the user component to produce a shared key. The gateway will compute the session key as follows:

```typescript
const isGroupChannel = msg.channelType === 'group';
const sessionKey = isGroupChannel
  ? `${msg.platform}-${msg.channelId}`           // shared across users
  : `${user.name}-${msg.platform}-${msg.channelId}`; // per-user (DMs)
```

The CCBuddy `sessionId` (used for message storage, memory context assembly) remains per-user via `buildSessionId()` — only the `sessionKey` for SDK session lookup uses the shared key for group channels.

## Components

### SessionStore (new — `packages/agent/src/session/session-store.ts`)

In-memory store mapping session keys to SDK session UUIDs. UUID generation is centralized here — callers never generate UUIDs.

```typescript
interface SessionEntry {
  sdkSessionId: string;   // UUID generated by SessionStore, passed to SDK
  lastActivity: number;   // Date.now() of last message
  isGroupChannel: boolean;
}

class SessionStore {
  private entries: Map<string, SessionEntry>;
  private timeoutMs: number;

  constructor(timeoutMs: number);

  /** Get existing session or create a new one (generating a UUID). Returns { sdkSessionId, isNew }. */
  getOrCreate(sessionKey: string, isGroupChannel: boolean): { sdkSessionId: string; isNew: boolean };

  /** Update lastActivity timestamp. */
  touch(sessionKey: string): void;

  /** Remove a session (e.g., on resume failure). */
  remove(sessionKey: string): void;

  /** Expire sessions older than timeoutMs. Called periodically. */
  tick(): void;
}
```

**Design decisions:**
- **In-memory only.** If Po restarts, session mappings are lost. Next message creates a fresh session seeded with memory context — graceful degradation with no data loss.
- **No persistence needed.** SDK sessions are stored on disk by Claude Code itself. We only need to remember the UUID mapping while the process is alive.
- **`tick()` called from the existing `setInterval` in bootstrap** (runs every 60s alongside `agentService.tick()`), keeping SessionStore passive.
- **UUID generation centralized here**, not in SdkBackend. `getOrCreate()` generates the UUID for new sessions; SdkBackend just uses the value it receives.

### AgentRequest Changes (`packages/core/src/types/agent.ts`)

Add one optional field:

```typescript
interface AgentRequest {
  // ... existing fields ...
  resumeSessionId?: string;  // SDK session UUID to resume (omit for new sessions)
}
```

Note: this is distinct from the existing `sessionId` field, which is the CCBuddy routing key (e.g., `dad-discord-dev`). `resumeSessionId` is the SDK's internal UUID for conversation continuity.

### AgentEvent Changes (`packages/core/src/types/agent.ts`)

Add SDK session UUID to the `complete` branch of the `AgentEvent` union so the gateway can confirm/store the mapping:

```typescript
// Existing pattern — inline union branches, not named interfaces:
export type AgentEvent =
  | AgentEventBase & { type: 'text'; content: string }
  | AgentEventBase & { type: 'complete'; response: string; sdkSessionId?: string }
  // ...
```

### SdkBackend Changes (`packages/agent/src/backends/sdk-backend.ts`)

```typescript
// In execute():
let sdkSessionId: string | undefined;

if (request.resumeSessionId) {
  // Resume existing session — do NOT set sessionId (mutually exclusive with resume)
  options.resume = request.resumeSessionId;
  sdkSessionId = request.resumeSessionId;
} else if (request.sdkSessionId) {
  // New session — use the UUID from SessionStore
  options.sessionId = request.sdkSessionId;
  sdkSessionId = request.sdkSessionId;
}

// ... existing query() call ...

// On complete event, include the sdkSessionId:
yield { ...base, type: 'complete', response: responseText, sdkSessionId };
```

Note: `request.sdkSessionId` is the UUID generated by `SessionStore.getOrCreate()` for new sessions. Add this as a second optional field on `AgentRequest`:

```typescript
sdkSessionId?: string;  // SDK session UUID for new sessions (generated by SessionStore)
```

### Gateway Changes (`packages/gateway/src/gateway.ts`)

In `handleIncomingMessage()`, compute the session key and conditionally assemble context:

```typescript
const isGroupChannel = msg.channelType === 'group';
const sessionKey = isGroupChannel
  ? `${msg.platform}-${msg.channelId}`
  : `${user.name}-${msg.platform}-${msg.channelId}`;

const { sdkSessionId, isNew } = sessionStore.getOrCreate(sessionKey, isGroupChannel);

// Only assemble memory context for new sessions — resumed sessions already have conversation history
const memoryContext = isNew ? assembleContext(user.name, sessionId) : undefined;

const request: AgentRequest = {
  // ... existing fields ...
  memoryContext,
  resumeSessionId: isNew ? undefined : sdkSessionId,
  sdkSessionId: isNew ? sdkSessionId : undefined,
};
```

After successful agent response:

```typescript
sessionStore.touch(sessionKey);
```

On agent error when `resumeSessionId` was set (resume failure):

```typescript
sessionStore.remove(sessionKey);
// Retry as new session with memory context
```

### Working Directory

The `workingDirectory` on `AgentRequest` is fixed for the lifetime of a session. Currently all requests use the same project root directory, and this does not change between messages. The SDK uses `cwd` as part of session context, so it must remain consistent across resumed calls.

### Config Addition (`packages/core/src/config/schema.ts` + `config/default.yaml`)

```yaml
agent:
  session_timeout_ms: 3600000  # 1 hour, configurable
```

Added to the config schema with a default of `3600000`.

### Bootstrap Wiring (`packages/main/src/bootstrap.ts`)

- Create `SessionStore` with `config.agent.session_timeout_ms`
- Pass it to the gateway
- Add `sessionStore.tick()` to the existing `setInterval` callback that calls `agentService.tick()` (runs every 60s)

## What Doesn't Change

- **Memory storage:** Messages are still saved to SQLite for long-term recall and consolidation.
- **CLI backend:** No session resumption — it remains the stateless fallback.
- **Scheduler/system requests:** Always one-shot with memory context. No continuity needed for briefings, heartbeats, etc.
- **Event bus architecture:** No new event types.
- **Activation mode logic:** Unchanged.
- **Memory consolidation:** Continues operating on SQLite message history as before.
- **DirectoryLock:** Continues to serialize concurrent requests to the same working directory. Sequential `resume` calls on the same SDK session are safe after the previous `query()` completes.

## Edge Cases

| Scenario | Behavior |
|----------|----------|
| Po restarts mid-session | Session store lost (in-memory). Next message creates fresh session with memory context. Graceful. |
| SDK session expired server-side | `resume` fails → catch error → `sessionStore.remove()` → retry as new session. |
| Very long sessions | SDK handles its own context window. 1-hour timeout naturally bounds most sessions. |
| User switches channels | Different sessionKey → different (or new) SDK session. Independent. |
| Multiple rapid messages | Same sessionKey → same sdkSessionId. `DirectoryLock` serializes execution so the previous `query()` completes before the next `resume` call begins. |
| System/scheduler requests | No `resumeSessionId` set → always one-shot. SessionStore not consulted. |
| `workingDirectory` change | Not expected — all requests use the project root. If it did change, it would be a new session key anyway. |
